02 — Core Algorithm

排班演算法

基於優先級的 Slot-Based 排班，確保高優先勤務先被填滿， 並透過跨週累積計數實現長期人員公平分配。

1

依優先級排序勤務，建立 Slot 清單

將所有勤務依 priority（1=最高）升序排列。每個勤務依 count × 工作天數 展開為 slot 清單。同一勤務的各輪天序隨機打亂，避免偏向固定某天。高優先勤務的所有 slot 一定排在低優先之前。

2

三層篩選可用人員

對每個 slot，從人員清單中篩除三類不可用者：① 當天有 offDays、② 本週累積班次達 maxShifts、③ 當天已被排過（防重複）。

3

三層排序決定最優人選

可用人員依以下優先序排序，取第一位：
① 跨週累積班次最少（長期公平）→ ② 本週已排次數最少（單週公平）→ ③ 技能分 × 0.6 + 隨機 × 0.4（偏好 + 不可預測性）

4

跨週累積追蹤

多週排班時，每週結束後將本週班次計數累加至 cumulativeShifts Map，下一週排班時作為第一優先排序依據，確保長期分配均勻。

// 1. 依優先級建立 slot 清單
const tasksByPriority = tasks
  .map((t, i) => ({ ...t, taskIndex: i }))
  .sort((a, b) => (a.priority || 9) - (b.priority || 9));

const slots = [];
for (const { taskIndex, count } of tasksByPriority) {
  for (let slotIndex = 0; slotIndex < count; slotIndex++) {
    const shuffledDays = [...workDays].sort(() => Math.random() - 0.5);
    for (const dayIndex of shuffledDays) slots.push({ dayIndex, taskIndex });
  }
}

// 2. 逐一填補 slot
for (const { dayIndex, taskIndex } of slots) {
  const available = personnel.filter(p =>
    !p.offDays?.includes(dayIndex) &&
    shiftCounts.get(p.name) < (p.maxShifts || 5) &&
    !assigned.has(p.name)
  );
  if (!available.length) continue;

  available.sort((a, b) => {
    // ① 跨週累積班次最少優先
    const cumDiff = cumulativeShifts.get(a.name) - cumulativeShifts.get(b.name);
    if (cumDiff !== 0) return cumDiff;
    // ② 本週已排最少
    const usedDiff = shiftCounts.get(a.name) - shiftCounts.get(b.name);
    if (usedDiff !== 0) return usedDiff;
    // ③ 技能分 × 0.6 + 隨機 × 0.4
    return getEffectiveScore(b, task.name) - getEffectiveScore(a, task.name);
  });
  assign(available[0], dayIndex, taskIndex);
}

04 — Engineering Decisions

技術決策

五個關鍵設計選擇，每一個背後都有明確的取捨理由。

📄
Vanilla JS，不用框架
系統只有單一頁面，無路由需求。純 HTML + Fetch API 減少打包步驟， 冷啟動時間為零，且伺服器可直接提供靜態檔案，無需 CDN 或 build pipeline。
🔌
Graceful DB Degradation
缺少 MONGODB_URI 時伺服器仍正常啟動，排班核心功能完全可用。 關注點分離：排班邏輯不依賴資料庫，資料庫只負責持久化。
⚖️
Slot-Based 而非逐日排班
傳統逐日排班會讓高優先勤務在某些天搶不到人。改為 建立全週 slot 清單再依序填補，保證優先級高的勤務先搶到資源，且各天機會均等。
🔒
API 邊界驗證
所有名稱欄位套用 /^[a-zA-Z0-9_\u4e00-\u9fa5-]{1,50}$/ Regex， 防止 NoSQL injection。不依賴前端驗證，後端是唯一信任邊界。
📊
fillStats 診斷回傳
每次排班後額外計算並回傳每個勤務的填補率（filled/needed）， 讓前端可以即時顯示哪些勤務排不滿，而不是讓使用者猜測。
🧠
預留技能分系統
taskScores 欄位已存入 schema， getEffectiveScore() 函數已實作， 但 UI 尚未提供輸入介面。向前兼容：資料模型先行，UI 後補。
05 — Testing & CI/CD

測試策略

使用 Jest + Supertest 對 API 層做整合測試，CI/CD 透過 GitHub Actions 自動執行。

測試涵蓋項目

✓
Health check — GET /api/status 回傳 200
✓
合法設定產生班表 — 完整流程端到端
✓
缺少必填欄位 — tasks / personnel 缺一回 400
✓
taskScores 驗證 — 超出 0–5 回 400，合法值通過
✓
Profiles API — 回傳 200 或 503（DB 離線）
✓
404 處理 — 未知路由回傳正確格式

CI/CD Pipeline

每次 push 到 develop 或 release，或開 PR 到 release 時自動觸發：

Checkout → Node 18 → npm ci → ESLint → npm test → index.html ✓ → Docker Build

CI 通過後，Zeabur 透過 Git 連線自動部署 release 分支，無需手動操作。

分支策略：feature/* → develop → release
